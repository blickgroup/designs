Design notes — Stage 3: Execution (LEFT-NAV variant, for comparison)

What this shows

Same project, same content, same side rail — but the tabs have been swapped for a left nav rail. Produced side-by-side with the tabs variant so Shaun can pick.

Why left-nav (the case for)

  • Scales to more sections without getting crowded — e.g. if we add Risks, Decisions, Budget later.
  • Sections stay visible always, so you can see at-a-glance that there are 8 tasks and 3 open follow-ups without clicking.
  • Mirrors app-level nav pattern (Compass lives in a left nav already), so nesting feels consistent.

Why NOT left-nav (the case against — why I lean tabs)

  • Eats ~220px of horizontal room that could be task detail. With a 320px right rail already, we're down to ~860px for the main column at 1400px max-width — task rows feel cramped on 13" laptops.
  • Double-chrome feeling: app nav + project nav + rail = three navigation surfaces. Tabs keep it to two.
  • Only 5 sections planned — tabs handle 5–7 beautifully, nav is overkill below 8.
  • Existing Compass mockups all use horizontal nav + section labels; tabs inherit that language.

Shown here

This mockup displays only the Tasks view to keep the comparison focused on nav chrome. Content/patterns (task+subtask cards, checkbox state, rail) are identical to the tabs variant.

Responsive behaviour

Below 1080px: left nav collapses to icons-only. Below 820px: left nav goes drawer (hamburger). Below 640px: drawer + rail stacked under main content. More brittle on mobile than tabs because we lose the nav context entirely in the drawer.

Tradeoffs

This is a near-mirror of tabs. Deliberate — the decision is chrome style, not information architecture. The right answer may depend on whether we plan to add more sections later (→ left nav) or stay lean (→ tabs).
