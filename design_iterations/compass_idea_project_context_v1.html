Design notes — Stage 1: Submitted / Voting (context only)

What this shows

This is today's pattern — ideas listed in a table, one row per idea. Shown for reference only so later stages can be understood in sequence. No redesign proposed here; the row → modal-become-page transition happens at stage 2 (planning). Clicking an approved row takes the user to the full-page project workspace in stage 2–4.

Layout choices

  • Table pattern preserved — Compass already uses rows-in-card for submitted items and it scales to ~50 ideas per cycle.
  • Status chip drives next action — Submitted = vote, Voting = vote more/approve, Approved = "Open project" CTA launches stage 2.
  • Vote count + pips mirror Compass execution card effort chip styling, reusing the 10px uppercase visual language.

Tradeoffs considered

  • Card grid vs table — rejected. Table compares vote counts and efforts at a glance; cards force scanning.
  • Inline expansion — rejected. Opening a project is a mode shift, not a peek; full-page is correct at approval time.

Responsive behaviour

Under 900px the vote column collapses into the title cell (secondary line), effort/submitted cells stack under title, CTA column stays rightmost. Field users on phone rarely submit — this is a desktop-weighted surface.

Open questions for Shaun

  • Should a "Voting closed — awaiting approval" intermediate state be visible, or does voting → approved happen atomically in cycle planning?
  • Who can click "Open project" for an approved idea — only the assigned owner, or anyone in the cycle?
