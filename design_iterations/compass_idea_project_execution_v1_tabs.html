Design notes — Stage 3: Execution (TABS variant, hero mockup)

What this shows

Project in active execution. This is the mockup that has to earn its keep. Everything populated: tasks with subtasks (collapsible), follow-ups, activity feed, docs, comments. Side rail shows live progress, owner, cycle and quick actions.

Tasks + subtasks — the new pattern

Indented list with chevrons, rendered as cards. Each parent task is a card row (checkbox + title + subtask-count pill + assignee + due date + menu). Click the chevron (or anywhere on the card head) to expand inline subtasks. Subtasks are 30% lighter: indent via padding-left, smaller checkbox, thinner row divider. Mirrors the follow-up subtasks checklist pattern ux-researcher flagged, so there's one mental model in Compass.

Why not a tree / kanban / nested cards: trees feel engineering-y, kanban pushes people to ship columns (we only need status via checkbox state), deeply nested cards waste horizontal room. Indented list + chevrons is the universal "I know what this is" pattern — think Linear, GitHub, Todoist. Checkbox state carries status (todo / in-progress / done / blocked) without a separate status column, keeping the row scannable.

Parent progress roll-up: when all subtasks are done, the subtask-count pill flips to success-green ("3/3 done"), the parent checkbox offers a one-click "mark complete" nudge. Never auto-complete — humans close parents.

Two-column layout

Left (1fr) = tabbed work area. Right (320px) = sticky context rail with status, key facts, linked docs, quick actions, related vault notes. Mirrors TicketDetailPage.tsx.

Tab ordering decision

Overview → Tasks → Follow-ups → Docs → Activity. Overview is a one-pager summary (brief + in-flight tasks + recent activity + docs preview) — we're showing it scrolled into its "active" state here with everything visible on one scroll. Tabs give deeper views. Counts on each tab give quick scannability ("oh, 3 open follow-ups").

Tradeoffs considered

  • Gantt/timeline as primary view — rejected. 8-person team, sub-10-task projects: Gantt is overkill. A target-start/target-end bar in the rail gives enough temporal context.
  • Separate "Status" column on task rows — rejected. Checkbox state already encodes status (todo / in-progress / done / blocked) with different fills. Saves a column, keeps rows scannable.
  • Comments as their own tab — rejected. Comments live under the activity feed in Overview — project-level discussion; per-task comments live in the task drawer (future).
  • Drag-to-reorder tasks — left for implementation; hover grip handle anticipated.

Responsive behaviour

Below 1080px: rail drops under main. Below 720px: task row grid collapses — assignee + due on a second line, subtasks stay indented. Tabs become horizontal-scroll. Field users viewing on phone get a read-first experience; edits work but the expectation is they're logging updates, not building plans.

Open questions for Shaun

  • Subtask checkbox states — is 4 states (todo / in-progress / blocked / done) right, or do we collapse to 3 (drop blocked, use a red flag icon instead)?
  • Do follow-ups live inside the project as proposed, or stay in the global follow-up KanbanView and just appear here as "linked"?
  • Should the activity feed merge task+follow-up+comment events, or split them?
