How an idea becomes a project

End-to-end map of an idea's journey — from submission through voting, allocation, planning, execution, and closure. Each stage shows the trigger to move, the actor, where the idea lives in the UI, and which mockup represents it. Stages 1 and 3 keep today's UI. Stages 2, 4, 5 and 6 are what we've been designing.

Edge case · Pulling an idea back from a plan
Once an idea is allocated to a cycle, it's a project. Going back to "just an idea in the bank" loses the trail of work already done. Two allowed paths instead:
Move cycle. Swap plan_id from C2 → C3. Workspace continues. Status stays as-is (planning or execution depending on where it was).
Park. Status flips to parked. Workspace becomes read-only. Can be un-parked later by re-allocating to a future cycle (status returns to in_planning). No "back to bank" path. Once committed — even partially — it's a project that's alive, parked, or done.
Edge case · GM rejects the close
Reject requires a comment explaining what needs to change — not just "no". Three valid reject reasons:
  • "Actually do the items you skipped" — owner ticks instead of skips, resubmits
  • "Your skip reason isn't good enough" — owner edits the reason, resubmits
  • "More work needed" — owner adds follow-ups, completes them, resubmits
On reject: status flips to in_execution, owner notified via follow-up with the reviewer's comment attached. When owner re-opens the close-gate modal, previous DoD state is pre-filled — they don't redo everything, just address the contested items. Round-trip continues until approved or owner parks the project.

Side systems the flow depends on

Definition of Done
Standard 4-item template applied at allocation: docs + files · Blickopedia · team comms · training. Owner can add project-specific items. Skipping any requires a recorded reason. Admin-editable globally.
Follow-ups (IAM)
Blick's polymorphic follow-ups table. Compass project creates them with entity_type='idea'. Review-on-skip fires a follow-up on the GM. Comms-on-close fires one on the owner. No new surface — reviewers act from their existing IAM inbox.
Entity tasks (subtasks)
Same polymorphic pattern. Each follow-up can have subtasks. Parent status rolls up but parent closes manually.
File attachments
Polymorphic file_attachments table. Updates, comments, and DoD evidence all attach via FileUploadInput.
GWS manager chain
Reviewer routing reads department_head_email + intermediate_manager_email from blick_identity.user_profiles. Fields already synced from Google Workspace but not yet exposed on users API — ~10-line backend change flagged.
Watercooler (future)
Closing comms field is Watercooler-ready. When it ships, closing announcements can auto-post. For now, owner gets a follow-up to post manually via Chat/email/standup.
