Design notes — Stage 2: Planning (v3 — activity-centred)

v3 changes from v2

  • Owner is assigned at allocation — shown in the meta strip, not editable by default. Small Reassign link visible only to admins (mocked as a muted, disabled-looking control with a role-gate label).
  • "Tasks" renamed to "Follow-ups" everywhere — matches hub-wide language (follow-ups = the unit of work in Blick). Tab, CTAs, empty-state, side rail, checklist.
  • Setup scaffolding collapsed. The "Get set up" empty-card grid and "Kickoff checklist" dominated v2. Those are first-day concerns. v3 collapses setup into a single dismissible banner with inline CTAs. The page is now lived-in: Activity feed + Documents own the main column.
  • Definition of Done — new. Every project gets the standard template (docs + files, process on Blickopedia, team comms, training). Owner can add project-specific items. Items must be ticked to close — or a reason-why-not must be recorded. Stops projects sliding to Done without closing the loop.
  • Related vault notes removed from this stage. Not useful in planning. Will appear on Execution/Done where the links have earned context.

Focus of the planning page

The earlier planning page was "scaffold a thing." This one says "manage a thing through planning." Setup lives as a banner you dismiss; the real surface is activity + docs + a clear Definition of Done to work toward. Matches how planning actually feels — you spend more time reading updates and filing docs than populating empty fields.

Definition of Done — template

Admin-editable template applied to every project on creation. Template items as of this mockup:

  1. Documentation + files filed (Drive + project docs tab)
  2. Process documented on Blickopedia
  3. Team communication sent (what shipped, what changed)
  4. Training delivered where required (or marked N/A with reason)

Owner can append project-specific items below the template. Closing a project = all ticked, or each un-ticked item has a reason recorded. Reason text appears on the Done page in the historical record so it's visible at retro time.

Owner reassignment permission

Reassign control is visible but role-gated. Non-admins see Owner as read-only. Admins see a small "Reassign" link. Tooltip for non-admins: "Contact an admin to reassign". Backend check on the API route, not just frontend hide — standard hub pattern.

Nav + layout

Tabs at top (Overview / Follow-ups / Docs / Activity) — "Tasks" tab removed, follow-ups is the unit. Two-column skeleton kept: main (~1fr) + sticky rail (320px).

Open questions for Shaun

  • Who counts as admin for reassignment — Shaun + partners only, or a broader admin role?
  • Is Blickopedia the actual system, or are we parking that name and using Drive/vault until we build it?
  • Definition of Done — can admins edit the template per cycle, or is it a global template edited rarely?
  • Should DoD reasons for skipped items require admin sign-off, or owner's word is enough?
