Design notes — Stage 2: Planning (v5 — updates composer + reviewer routing)

v5 changes from v4

  • Updates / comments composer under Brief. The team's daily surface for notes, file attachments, and URLs. Uses the FileUploadInput pattern (dashed drop zone, pill cards for attached files) and --chart-N avatar rotation so it looks and feels like the rest of the hub. Populated with two example updates so you can see the shape.
  • Close-gate modal now includes "How is this being communicated?" field. Owner writes the closing message inside the modal. On close, this text saves as the project's closing announcement and auto-creates a follow-up on the owner to actually post it (Chat, email, all-hands — their call). No phantom Chat-bot integration. Watercooler-ready: when it ships, the announcement feed seeds from here.
  • Reviewer banner, dynamic. When any DoD item is skipped, the modal footer shows the named reviewer ("Shaun Tubb — GM Operations — will review before close"). Updates live as items flip between ticked and skipped. Zero skips = direct close, no review.
  • Reviewer resolution algorithm — role-based. Not a shared admin pool. Lookup walks the owner's manager chain in Google Workspace: 
    reviewer(owner):
  if owner == Tim:   return Shaun
  if owner == Shaun: return Tim
  walk manager chain from owner
    — first person found who is Tim or Shaun → reviewer
    — if chain exhausted / broken / null → Shaun (fallback)
    Self-approval prevented. Sales team (Tim's reports) walks up to Tim (GM Sales); everyone else walks up to Shaun (GM Operations).
  • Admin review is a follow-up, not a new surface. If any items skipped, project enters Awaiting admin review state. A single review follow-up fires on the routed GM — lives in their existing IAM follow-ups inbox. Approve → project closes, comms follow-up fires on owner. Reject with comment → owner addresses, resubmits.

Component vocabulary used

  • Comment composer: greenfield — no canonical CommentInput exists today (just textarea + post button in IdeaDetailModal). v5 composes from the FileUploadInput shape + existing input styling. Flagged as build-time opportunity: promote this pattern to a shared component.
  • Attachments: FileUploadInput pattern — dashed border drop zone, pill cards for attached files, file icon + filename + size + remove ×.
  • Follow-up card: mirror FollowUpDetailPanel — assignee avatar + title + status chip + due date.
  • Status chips: canonical hub vocabulary — success-bg/text for done, info-bg/text for in-progress, warning-bg/text for todo, bg-tertiary for cancelled.
  • Avatar: --chart-1 to --chart-8 rotation via name hash. 26–28px in card rows, 22px in rails, 40px in modal headers.
  • Modal shell: ModalShell proportions — ~680px max-width, header+body+footer with borders, 20px/24px padding, overlay uses rgba dimming.

Backend dependency — small

The user lookup API at services/operations-api/index.js:254 currently SELECTs email, full_name, country only. The BigQuery blick_identity.user_profiles table already has department_head_email + intermediate_manager_email columns synced from Google Workspace. We need to expose these two columns on the /api/health-safety/users response and add to UserRecord type + Zod schema. ~10 line change. No new table, no migration, no new data source. Routing logic is pure frontend on top of those two fields.

Edge cases handled in reviewer routing

  • Owner = Tim → Shaun reviews (mutual escalation)
  • Owner = Shaun → Tim reviews (same)
  • Owner has no manager in GWS → Shaun (default fallback)
  • Owner's manager is not Tim or Shaun (e.g. senior engineer managing a junior) → walk up the chain until Tim or Shaun is found

Open questions for Shaun

  • Does the reviewer see the skipped-item reasons inline in their follow-up, or do they click through to the project Done preview to see them in context?
  • Should "reject" require a comment from the reviewer (my lean: yes — silent rejections leave owner guessing)?
  • Is the comms follow-up auto-created at approval, or at the moment the owner submits (even if pending review)?
  • When Watercooler ships, should closing announcements post automatically to it, or still require the owner to do it manually via their comms follow-up?
